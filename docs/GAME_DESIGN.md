# Apex-arena — 게임 설계 문서 (v0.3)

> 자율주행 알고리즘을 **쉽게 배우고 + 경쟁하는** 브라우저 레이싱 게임.
> 오리지널 트랙 · 탑다운 2.5D(slant+**높낮이**) · **온로드+오프로드** · 디자인 최우선.

| 축 | 결정 |
|---|---|

## 1. 한 문장 컨셉

센서가 들어오면 조향·속도를 뱉는 **'차의 두뇌'**를 짜서, 트랙에서 랩타임과 배틀로 겨루는 게임.
실제 AV 스택 구조 **인지 → 계획 → 제어**를 그대로 게임 규칙으로 사용.

## 2. 설계 급소: 하나의 결정론 sim + 하나의 두뇌 계약(Contract)

세 티어와 세 경쟁 모드가 **전부 같은 엔진·같은 인터페이스** 위에서 돌아야 재작성이 없다.

### 2.1 두뇌 계약 (모든 티어 공통)
```
step(observation) -> action

observation = {
  scan:      float[N]      # LiDAR 거리 (예: 이젤 108빔), 결정론
  pose:      {x, y, yaw}   # 트랙 좌표계
  vel:       {vx, vy, w}   # 차체 속도
  progress:  float         # 트랙 진행률 s
  grade:     float         # 진행방향 종단경사(rad, +오르막)  ← 지형
  bank:      float         # 횡단경사/뱅킹(rad, +유리방향)    ← 지형
  z_ahead:   float[K]      # 진행방향 전방 고도 프리뷰(크레스트 예측용)
  # (선택) 상대차: opponents[] — 배틀 모드에서만 채워짐
}

action = {
  steer:    float  # [-1,1] → 조향각으로 매핑
  throttle: float  # [-1,1] → 가/감속
}
```
- **티어1** 슬라이더도, **티어3** 파이썬 코드도 결국 이 `step()` 하나를 구현/파라미터화하는 것.
  세 티어는 "이 함수를 어떤 인터페이스로 채우느냐"의 차이일 뿐.

### 2.2 물리엔진 (결정론이 생명)
- **자전거 모델 + friction ellipse (ggv)** — 님의 kinematic+ggv 그대로.
  (참고: `[[project_ifac2026_model_choice]]` — 저그립 우레탄 노면 = kinematic+friction ellipse 기본)
- **고정 timestep**, 난수·wall-clock 금지 → 같은 입력 = 같은 리플레이 = 공정한 리더보드.
- 충돌·트랙이탈 판정 명확화 (벽 = 페널티/리셋).

#### 2.2b 지형 높낮이 = 준3D (높이필드 z(x,y), 셰이딩 아님)
평면 상태(x,y,yaw,v)는 유지하되, **정적 높이필드 z(x,y)** 를 샘플해 힘을 변조. 3D 차량동역학 아님 → 결정론·경량 유지. 렌더와 물리가 **같은 z 하나**를 소스로 씀(불일치 없음).
- **종단경사(grade)** → 중력 종방향 성분 `a = -g·sin(θ_grade)`: 오르막 감속, 내리막 가속.
- **하중 변조(normal load)** → `N = m·g·cos(θ)` (±수직곡률: 크레스트=하중↓, 딥=하중↑). friction ellipse 한계가 `F_max = μ·N` 로 스케일 → **크레스트에서 그립 예산 감소, 딥에서 증가.**
- **뱅킹(bank)** → 코너 안쪽으로 기운 노면은 가용 횡력↑(뱅크드 오벌), off-camber는 횡력↓.
- **에어본(옵션 플래그)** → 급크레스트+고속으로 `N≤0`이면 착지까지 타이어힘 0. *스킬·손맛↑이지만 스코프↑ → MVP 이후 플래그.*
- 알고리즘적 배당: 컨트롤러가 **경사·하중을 예산에 반영한 속도 프로파일**을 배워야 함(실제 3D 레이싱라인 최적화와 동형). μ는 §3.5 오프로드에서 지역가변까지 확장.

### 2.2c 제어 메커니즘 확장 로드맵 (Pure Pursuit 기반 → Stanley·MPCC·MPPI·RL)
**불변 계약**: 모든 메커니즘은 결국 `step(obs)→(steer,throttle)` 하나로 수렴(§2.1) → 코어·물리·리플레이 재작성 불필요. 달라지는 건 **파이프라인 분해를 얼마나 걸치느냐**뿐.

| 메커니즘 | 스택에서 차지하는 범위 | 웹 구현 난이도 | 킬러 시각화 |
|---|---|---|---|
| **Pure Pursuit** (기반) | Control | 완료 | L1 점·조향 |
| **Stanley** | Control | 완료(쉬움) | cross-track+heading 오차 |
| **MPPI** | **Planning+Control 융합** | **쉬움·웹친화** — 내 결정론 자전거모델을 그대로 N개 롤아웃, 코스트 가중평균 | **샘플 롤아웃 구름**(수백 궤적) → 매우 시각적 |
| **MPCC** | Planning+Control 융합 | 중(경량 QP/NLP를 JS로, 소형 horizon) | 예측 horizon(실스택 `/mpc_trajectory/markers`) |
| **RL 정책** | **전체 스택 end-to-end 가능** | 중~상 (오프라인 학습→가중치 임베드 추론, 또는 Pyodide 티어서 학습) | 정책 attention·value·롤아웃 |

**구조적 함의**: 5단계 분해(§3.5)는 고전 분해형(pursuit/Stanley)엔 딱 맞지만, MPCC/MPPI는 **Planning·Speed + Control을 한 블록이 통째로 대체**(speed 블록 그레이아웃), RL은 **여러 단계를 통째로 대체하는 end-to-end 블록**. → CONTROL 블록이 "speed까지 접수하는 통합 컨트롤러" 플래그를 갖도록 설계(현재 compile()에서 steer/targetV 분리 → 통합모드 소량 리팩터로 수용). **다음 확장 1순위 = MPPI**(내 sim 모델 재사용 + 샘플궤적 시각화가 압도적). 모든 확장은 Pure Pursuit 대비 벤치마크로 붙음.

### 2.3 왜 이게 급소인가
- 이 계약을 MVP 때 확정하면, 티어2/3와 배틀은 **엔진 재작성 없이 인터페이스만 추가**.
- 리더보드 치팅 방지도 여기서 공짜로 나온다 → §5.

## 3.6 조립 모델 (Composition Model) — "유저가 어떻게 알고리즘을 구성하는가" (확정 2026-07-14)

결정: **백본 = 스테이지 레인 + 걸치는(spanning) 블록** / **저작 깊이 = 프리셋+파라미터+조합**(코드는 나중). 지금 6-블록은 이 모델의 "레인당 1블록·프리셋+파라미터" 특수 케이스.

### 3.6.1 백본: 5 레인, 블록이 걸칠 수 있음
```
Localization │ Perception │ Detection │ Planning │ Control
 GT/Odom     │ Map/LiDAR  │  None/Obj │ line/speed│ PurePursuit  ← 고전: 레인당 1블록
             │            │           │ [    MPPI/MPCC    ]      ← Plan+Control 걸침(speed 그레이아웃)
 [               RL policy: Percep→Control 통째로              ]  ← end-to-end
```
멘탈모델(항상 5 관심사) 유지 + 모든 알고리즘군 수용. 자유 노드그래프는 기각(위압적).

### 3.6.2 스테이지 간 계약 (5개 — 조립의 뼈대, 시각화 대상과 동일)
| 경계 | 타입 `contract` | 필드(2D sim 구체) |
|---|---|---|
| Localization → | **Pose** | `{x,y,yaw, cov}` (truth 또는 노이즈 추정) |
| Perception → | **World** | `{ref: 전방 센터라인 s·curvature·width, edges: 좌우경계, gaps: free-space rays}` (map=정확 / lidar=노이즈) |
| Detection → | **Objects** | `[{x,y,vx,vy,size,isStatic}]` (빈트랙=[]) |
| Planning → | **Reference** | `{path: waypoints, speed: 지점별 목표속도}` |
| Control → | **Command** | `{steer∈[-1,1], throttle∈[-1,1]}` → 물리엔진 |
| (원천) 센서 | **Sensors** | `{scan: LiDAR, imu: r·accel, odom: wheel v}` — Perception/spanning이 소비 |

### 3.6.3 블록 인터페이스 (선언형)
블록이 선언: `lanes`(점유 레인, 연속) · `consumes`(진입경계 계약 + 필요시 Sensors) · `produces`(출구경계 계약) · `params` · `compute(in,params,sensors)→out` · `viz`(시각화 노출: lookahead점·rollout구름 등).
컴파일러가 레인을 훑어 배선: 각 블록은 직전이 생산한 계약을 소비, spanning은 진입계약→출구계약(중간 스킵). **불변 최종 계약 = Command `(steer,throttle)`** = §2.1.

### 3.6.4 조합(Compose) 깊이 = 코드 없이 "알고리즘 구성"
레인이 블록 1개를 넘어 **결합자**를 담음:
- **스위치/행동규칙**: "평소 Racing-line, 장애물 X 이내면 Gap-follow" (실스택 state machine: GB_TRACK/OVERTAKE/TRAILING/FTG).
- **블렌드**: 가중 혼합 (steer = w·pursuit + (1-w)·gap).
→ 실차 behavior layer를 코드 없이 조립.

### 3.6.5 유효성
컴파일러 검사: 모든 레인 정확히 1회 커버(단일/spanning) + 계약 정합(Control-only는 Reference 필요, 없으면 기본 Planning 자동삽입 or 에러). 무효 구성은 UI에 표시.

### 3.6.6 진행/개방
초반 상류 레인(Loc/Percep)은 이상화 프리셋으로 잠금→Control 집중. 진행하며 상류 레인·조합·(먼 미래)코드 깊이 개방. spanning 블록(MPPI→MPCC→RL)은 상위 언락.

### 3.6.8 ★확정 조립 UX = ROS 네이티브 데이터플로 그래프 (2026-07-15, 앞 모델 대체)
앞선 모듈-레인/Blockly 모델은 "고르기=튜닝"이라 거부됨. 확정=**파인그레인 데이터플로 그래프**(노드=ROS노드, 와이어=타입토픽). 목업 design/graph_mockup.png. 원칙: **아무것도 강제 블랙박스 아님.**

**2계층 노드 + 캡슐화 (유저가 노드를 만든다):**
- **Primitives**: 원자. 수학(+−×÷,atan2,abs), **배열/반복(Map·Reduce·Slice·Window·Argmax·ZipWith)**, 기하(distance,angle,유한차분,투영,보간/스플라인), 구조체접근(wpt.x).
- **Composites**: 프리미티브(또는 합성)를 묶은 **서브그래프**. Curvature ahead·Lookahead Point·Pure Pursuit·Racing-line gen = 전부 합성 = 열어서 fork 가능.
- **캡슐화**: 서브그래프 선택→"노드로 만들기"→포트 지정→재사용 노드(Blueprints collapse / Houdini subnet / LabVIEW subVI / ROS pkg 동형).
- **프리셋 = 열리는 합성**(오픈 서브그래프). **개인 라이브러리 + 공유**(while True 커스텀노드, ROS pkg처럼).

**이걸 가능케 하는 핵심 = 리치 타입 + 반복**: `num·vec2·angle·pose·array<T>·waypoint{x,y,s,κ,width}·scan[]` + Map/Reduce/Window/Slice. ← 커스텀 노드의 토대(진짜 작업).

**예시(짓기)**: Curvature ahead = waypoints→Slice(idx,W)→Map(heading)→Diff, ArcLengths→Divide→Max|abs|→κ, 선택후 "노드로 만들기". Lookahead=Advance(idx,Ld/ds)→Index, 안쓰면 안만듦(=설계선택). Waypoint생성도 그래프(센터라인=Bounds→Map(midpoint), 레이싱라인=min-curv step을 Loop).

**다음**: (1)프리미티브·타입세트 스펙확정, 또는 (2)"합성 열기/커스텀 만들기" 플로우 목업. 노드 입자 크기(순수수학 vs 기하단위) 결정 필요.

### 3.6.7 "블록 코딩" UX — 레퍼런스 기반 형태 (조사 2026-07-14, 3.6.8로 대체)
사용자 방향="블록 코딩 느낌". 조사 결과 우리 빌더 = **2단 구조**:
- **매크로(스택 파이프라인)** = *while True: learn()* 식 노드 파이프라인. 5 레인 블록이 **타입 포트(=§3.6.2 계약)** 로 스냅, 데이터 아래로 흐름. 걸치는 블록=한 노드가 여러 레인 덮음. *Scratch/Blockly* 식 **모양 커넥터 → 안 맞는 연결 불가**(유효성 시각화).
- **마이크로(Planning/행동 블록 속 = Compose §3.6.4)** = *Gladiabots* 식 **if-then + 우선순위**(또는 소형 behavior tree). 코드 없이 행동 프로그래밍.

레퍼런스 훔칠 것:
| 소스 | 훔칠 것 |
|---|---|
| **Gladiabots** | if-then+우선순위 규칙 조립(코딩0) · **비동기 ELO 대전**(동시접속 불필요) → 리더보드/배틀 |
| **while True: learn()** | 노드+데이터흐름 · **각 블록이 실제 기법+현실 설명 가르침** · compute 예산=다중지표 |
| **Scratch/Blockly/mBlock** | 끼워맞춤 유효성 · 레인별 색 · **블록↔Python 토글(같은 프로그램 두 뷰)** = 저작사다리 |
| **Behavior Trees**(Unreal/Godot) | 조합 레이어 파워유저 옵션 |

원칙: **블록 = 곧 `step(obs)→action`**(§2.1). 지금 블록, 나중 Pyodide가 동일 프로그램을 Python으로. 각 블록에 "실제 로보틱스에서 뭔지" 설명 부착(학습 목표).

## 3.5 도메인 축: 온로드 × 오프로드 (티어 축과 직교)

두 도메인은 **같은 결정론 엔진(§2.2) · 같은 두뇌 계약(§2.1)** 위에서 돈다. 재작성 없음 — 다른 건 파라미터·트랙 시맨틱뿐. 이렇게 하면 매트릭스가 **2 도메인 × 3 티어**로 콘텐츠가 두 배가 되고, 실제 연구 두 갈래(온=F1TENTH/트랙레이싱, 오프=traversability/rally)와 맞물린다(`[[project_offroad_racing]]`).

| 구분 | 온로드 | 오프로드 |
|---|---|---|
| **그립(μ)** | 높고 균일 | **낮고 지역가변**(흙/자갈/진흙/눈) → 슬립·드리프트 |
| **경계** | 하드월·커브(벽=충돌) | 소프트 경계 + traversability(러프니스 페널티, 하드월 적음) |
| **관측 추가** | — | scan에 **표면 그립/지형타입** 힌트 |
| **목표** | 정밀 최소랩타임 | **강건 완주 + 슬립 관리** (`[[feedback_robustness_over_speed]]`) |
| **가르치는 것** | 레이싱라인·정밀추종(pure pursuit/MPCC) | friction-limited 제어·드리프트·지형인지 계획 |
| **비주얼** | 쿨한 아스팔트, 크리스프 커브 | 웜톤 대지·먼지 파티클·타이어 자국 (Art of Rally 톤) |
| **지형(높낮이)** | 뱅크드 코너·완만한 언덕·크레스트(정밀 하중관리) | 급경사·점프·off-camber(에어본·강건성 시험) |

**엔진 선반영 원칙:** 물리엔진을 처음부터 **가변 friction ellipse(지역별 μ 맵)** 로 짜두면, 오프로드는 μ 낮추고 지형 레이어만 얹는 콘텐츠 추가가 된다. **MVP는 온로드만**(단순 그립으로 '재미' 먼저 검증), 오프로드는 엔진·티어1 검증 후 도메인으로 투입(§7 Phase 2.5).

## 4. 콘텐츠: 학습 코스 (티어1이 곧 튜토리얼)

님의 알고리즘 자산이 곧 레벨 정답지 + 커리큘럼. **온로드 트랙**:

1. **직관 잡기** — lookahead 슬라이더로 언더/오버스티어 체감 (pure pursuit)
2. **속도 프로파일** — 코너 감속 곡선 튜닝, friction 한계 체험
3. **반응형 회피** — Follow-The-Gap(FTG) 파라미터 (`[[project_men472_practice_format]]` Week6-1 FTG 재활용)
4. **경로 최적화** — 레이싱라인 vs 센터라인
5. **(티어3) 직접 구현** — 위 개념을 파이썬 `step()`으로

**오프로드 트랙** (별도 커리큘럼 브랜치):
1. **그립 감각** — 저μ 노면에서 왜 미끄러지나, throttle 관리
2. **드리프트 제어** — 카운터스티어로 슬립각 잡기
3. **지형 읽기** — 자갈/진흙 회피 vs 관통(traversability 트레이드오프)
4. **강건 라인** — 최속보다 안 튕기는 라인(대회 감각)

각 레벨: **개념 설명 → 슬라이더 놀이 → "이 랩타임/완주 깨보기" 챌린지** 3단.

| 단계 | 모드 | 물리 복잡도 | 도입 시점 |
|---|---|---|---|

### 순위 UX = 리그 사다리 + 히스토그램 (조사 반영)
- 순수 글로벌 순위표 대신 **리그 보스봇 승급 게이트**(CSB) — 이기면 다음 리그 개방 = 온보딩·매치메이킹 동시 해결.
- 점수는 **히스토그램 + 다중 적대 지표**(Zachtronics): *랩타임 / 일관성 / 무충돌* — 내 점수를 분포 속에서 보여줘 "1등 아니면 패배" 압박 제거.
- 솔로 진행은 **트랙별 메달 티어**(Trackmania): 브론즈/실버/골드/개발자.

### 치팅 방지 (결정론의 배당금)
- 클라이언트는 **입력 시퀀스 + 결과 리플레이**만 제출.
- 서버가 같은 결정론 엔진으로 **재검증** → 랩타임 위조 불가.
- 고스트·리더보드는 리플레이 저장만으로 구현 (영상 저장 불필요).

Apex-arena는 사실상 **"DeepRacer의 리그를, 느린 RL 학습 대신 즉각·결정론·고전 알고리즘으로"** 하는 포지션.

| 게임 | 훔칠 것 (Steal) | 피할 것 (Avoid) |
|---|---|---|
| **CodinGame — Coders Strike Back** (2D 체크포인트 pod 레이싱, 12만 참가) | **리그 사다리 + 보스봇 승급 게이트**: 순수 리더보드 대신 "리그 보스 AI를 이기면 승급". 온보딩+매치메이킹 동시 해결. / 정확한 결정론 물리를 유저에 공개 → 고수는 genetic·MC 탐색까지 감 | 초반부터 2 pod 협동 등 복잡 규칙 노출 |

| **Trackmania** (고스트/리플레이/메달) | **트랙별 메달 티어**(브론즈/실버/골드/개발자): 글로벌 1등 없이도 솔로 목표. 고스트 오버레이가 핵심 손맛 | — |
| **Art of Rally / Absolute Drift** (탑다운 2D, 핸들링 강박 튜닝) | 탑다운 2D가 프리미엄일 수 있음 증명. **약간 기울인(2.5D slant) 뷰**가 정사영보다 잘 읽힘. 핸들링 '느낌'에 투자 | 순수 정사영의 밋밋함 |
| **AI Learns To Drive** (NEAT 진화 자동차, Steam) | 시장 수요 존재 증명 | 블랙박스 NN 진화. 우리는 **이해·손튜닝 가능한 고전 알고리즘**이 차별점 |

### 이 조사가 바꾼/굳힌 결정
1. **경쟁 = 리그 사다리 + 보스봇 승급** (CSB). 순수 랩타임 순위표 대신, 각 리그 보스 AI를 이겨 승급. §5 갱신.
2. **점수 = 히스토그램 + 다중 적대 지표** (Zachtronics). 랩타임 하나가 아니라 *속도/일관성/무충돌* 여러 축 → 결정론 재검증과도 궁합. 글로벌 1등 압박 완화.
3. **트랙별 메달 티어** (Trackmania). 솔로 진행 루프 → **MVP에 포함**하면 "0.2초 더" 욕구를 즉시 제공.
4. **뷰 = 약간 기울인 2.5D slant 탑다운** (Art of Rally). 순수 2D 확정하되 살짝 기울여 가독성↑.
5. **포지셔닝 = "즉각·투명한 DeepRacer"**. 우리의 킬러는 슬라이더 밀면 궤적이 실시간으로 사는 것.

## 6.6 아트 디렉션 & 디자인 시스템 (최우선)

> 목표: **"어색하지 않고 깔끔하게 예쁨."** 깔끔함은 화려함이 아니라 **절제**에서 나온다 — 제한된 팔레트, 넉넉한 여백, 액센트 하나, 일관된 규칙.

### 6.6.1 아트 디렉션 한 줄
**미니멀 플랫 벡터 + 살짝 기울인 탑다운 + 테크니컬/에디토리얼 정제감.**
레퍼런스: *Absolute Drift*(미니멀) × *Art of Rally*(따뜻함·읽힘) × 프리미엄 텔레메트리 툴(데이터의 우아함). 스프라이트 픽셀아트 아님 — 깔끔한 벡터 도형 + 결(texture)은 아주 은은하게.

### 6.6.2 컬러 시스템 (핵심: 저채도 베이스 + 액센트 1개)
- **베이스는 뉴트럴**: 라이트=오프화이트/웜그레이, 다크=딥차콜. 채도 거의 0.
- **액센트는 플레이어 차 딱 하나** — 여기에만 비비드 컬러 몰아줌(눈이 항상 내 차를 찾음).
- **도메인으로 무드 스왑**(같은 골격, 다른 팔레트):
  - 온로드 = 쿨(아스팔트 블루그레이 + 시안/화이트 커브)
  - 오프로드 = 웜(대지 탄/오커 + 먼지)
- **의미색은 절제**: 최속 라인=은은한 골드, 고스트=반투명, 충돌=낮은채도 레드. 신호등처럼 원색 남발 금지.
- 라이트/다크 둘 다 1급 지원(님 아티팩트 습관과 동일).

### 6.6.3 트랙·차량 렌더 규칙
- 트랙: **플랫 필 + 아주 긴 은은한 롱섀도우** 하나로 slant 깊이감. 노이즈/그라디언트 최소.
- 커브(연석): 깔끔한 적/백 대시, 1~2px 크리스프.
- 레이싱라인: 얇은 1px, 골드 반투명. 텔레메트리 궤적: 속도로 색 매핑(느림→빠름).
- 차: **단순 라운드 실루엣** + 액센트색. 바퀴 스프라이트 대신 **미세 트레일·타이어자국**으로 속도감.
- 오프로드 먼지: 파티클 소량, 저알파. "많이 = 지저분"이 됨, 절제.

**높낮이(2.5D) 렌더 규칙** — 물리와 같은 z(x,y)를 소스로:
- **힐셰이드**: 고정 광원 기준 경사·향(aspect)으로 은은한 명암 → 기복이 읽힘(등고선 남발 금지, 셰이딩으로).
- **스크린 Y 오프셋**: 고도에 비례해 트랙·차를 화면 위로 밀어 slant 입체감.
- **차 드롭섀도우**: 높이에 따라 그림자 간격↑, **에어본 시 간격이 확 벌어짐**(고전 2.5D 높이 신호).
- **뱅킹**: 노면 크로스섹션 명암·커브를 기울여 표현.
- 원칙은 여전히 **절제** — 기복은 그림자·명암으로, 텍스처 떡칠 금지.

### 6.6.4 UI / HUD (여기서 '싸구려 느낌' 갈림)
- **8px 스페이싱 그리드 + 라운드 스케일(4/8/12) 고정.** 아무 데나 여백 금지.
- 타이포: UI=기하학적 산세리프 1종, 텔레메트리·코드=모노 1종(테크니컬 신뢰감). 타입스케일 5단 이하로 타이트.
- HUD는 **데이터비주얼 품질**: 속도/조향 트레이스, 델타바, 스플릿 — dataviz 스킬 원칙 적용(님 프로젝트 표준). 어두운 사각형+흰글자 남발 금지(`[[project_pokemon_mmo_ui]]` 교훈과 동일 철학).
- 모션: 카메라 미세 랙, 이징(ease-out), 고스트 페이드. **절제된 저크(juice)가 프리미엄**.

### 6.6.5 디자인 시스템 산출물 (코드 전에 만든다)
1. **스타일 타일(style tile)** — 팔레트·타이포·차·트랙 한 조각·HUD를 한 화면에. *비인터랙티브 목업 = 게임 코드 아님.*
2. **토큰 파일**(`design/tokens.ts`) — 색/간격/라운드/타이포를 한 곳에서. 온/오프로드 팔레트 스왑도 여기서.
3. 이 둘을 **Phase 0보다 먼저** 확정 → 이후 모든 렌더가 토큰만 참조.

## 8. MVP 상세 스펙 (Phase 0+1)

**목표:** 한 트랙에서 슬라이더로 pure pursuit를 튜닝해 랩타임을 깎는 게 *재미있는가*.

- [ ] 결정론 자전거+ggv 물리 (고정 dt)
- [ ] **높이필드 z(x,y) + 경사·하중 변조** (§2.2b) — 렌더/물리 공용 소스
- [ ] 트랙 1개 (중심선 + 벽 + **고도** + 뱅크드 코너, 이탈 판정)
- [ ] LiDAR scan 생성 (N빔 레이캐스트)
- [ ] 두뇌 계약 `step(obs)->act` 확정 + 내장 pure pursuit
- [ ] 2D 탑뷰 렌더 + 실시간 궤적/HUD(속도·랩타임)
- [ ] 티어1 슬라이더: lookahead, 목표속도, 게인
- [ ] 랩타임 측정 + 베스트 고스트 리플레이 (로컬 저장)
